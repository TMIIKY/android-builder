二、严重缺陷 (P0 - 可导致数据丢失或功能崩溃)

P0 #1：备份导出不包含 IndexedDB 正文数据
根因：DataBackup.collectAllData() 仅收集 localStorage 中的数据（通过 key 前缀匹配），而章节正文实际存储在 IndexedDB 中。Storage.saveChapters() 会将 content 字段从元数据中剥离（替换为 _hasContent 标记）。
影响：用户导出备份后，如果清空数据再导入，所有章节正文会丢失（元数据恢复但正文为空）。
文件：data-backup.js:299-326 (collectAllData), storage.js:433-453 (saveChapters)
✅ 建议：在 collectAllData 中增加 IndexedDB 数据收集，或实现 exportAll() 方法将 IndexedDB 章节也打包到备份中。
P0 #2：generateChapters() 中 saveChapters 使用异步版本，未等待 IndexedDB 写入
根因：app.js:727 行调用 Storage.saveChapters(bookId, existingChapters)（异步版本），紧接着就开始 AI 生成流程。如果此时发生异常或 App 崩溃，空壳章节已创建但正文未写入 IndexedDB。
文件：app.js:727
✅ 建议：改用 await Storage.saveChaptersSync() 确保空壳章节写入完成后再开始 AI 生成。
P0 #3：updateChapterTitle() 使用异步 saveChapters，可能导致标题丢失
根因：app.js:1316 行用户编辑章节标题后调用 Storage.saveChapters()（异步），未等待 IndexedDB 同步。如果用户快速切换到其他页面，元数据更新可能未完成。
文件：app.js:1310-1319
✅ 建议：关键写操作统一使用 saveChaptersSync。
P0 #4：deleteChapter() 删除章节但未同步清理 IndexedDB
根因：app.js:1331 行调用 Storage.deleteChapter()，但该方法内部只调用 saveChapters()（异步），未显式清理 IndexedDB 中的章节正文。如果 deleteChapter 方法只更新了 localStorage 元数据，IndexedDB 中的正文不会被删除。
文件：storage.js (deleteChapter 方法), app.js:1331
✅ 建议：在 Storage.deleteChapter 中显式调用 ChapterDB.deleteContent() 清理 IndexedDB。

三、高风险缺陷 (P1 - 影响用户体验但非致命)

P1 #5：localStorage 容量溢出风险 — 大规模数据时备份/导出失败
根因：localStorage 有 5-10MB 上限。当用户创作大量书籍（如 20 本 × 500 章），元数据（chapters_*、memory_*、plans_*）累积可能接近上限。storage.js 虽有 _emergencyCleanup 机制但仅删除 reviews_ 和 specs_，释放空间有限。
影响：写操作静默失败、自动备份因数据过大被跳过、用户操作丢失。
文件：storage.js:261-286 (_emergencyCleanup), storage.js:200-209 (_triggerAutoBackup)
✅ 建议：增加存储用量监控面板；将 chapters 元数据也迁移到 IndexedDB；定期压缩 memory 中的冗余数据。
P1 #6：AI 调用无重试机制 — 网络波动时单次失败即终止
根因：AIService.callAPI() 没有内置重试逻辑。AI 生成章节（generateChapters）循环中，一个章节失败后直接跳过继续下一个，但如果是临时网络抖动，重试可能成功。
文件：ai-service.js:91-132 (callAPI), app.js:915-919 (catch 块)
✅ 建议：在 callAPI 中增加指数退避重试（最多 3 次），只对网络错误和 5xx 重试，4xx 不重试。
P1 #7：buildContextMemory 在长篇小说中可能超出模型上下文窗口
根因：虽然 buildContextMemory 已做了分层截断（近期完整、前期采样），但对于 500 章的小说，聚合后的 context 仍可能达到 10K+ tokens。加上前文结尾 500 字 × 3 章 + Spec 1500 字 + System Prompt 3K 字，总计可能超过 20K tokens。
文件：storage.js:778-954 (buildContextMemory)
✅ 建议：在 buildContextMemory 中增加 token 估算和硬截断；提供"压缩模式"选项。
P1 #8：generateChapters 并发生成无节流控制
根因：app.js:737-920 行用 for 循环逐个生成章节，但每次调用 AI API 后立即进行记忆更新、矛盾检测。如果用户选择生成 5 章，整个过程可能耗时 5-10 分钟。期间如果网络中断或 API 限流，后续章节全部失败。
文件：app.js:737-920
✅ 建议：在每次 AI 调用之间加入最小间隔（如 1 秒）；增加失败重试次数（如每章最多重试 2 次）。
P1 #9：脚本加载顺序依赖 — 单个模块加载失败导致全局崩溃
根因：index.html:391-402 行按顺序加载 11 个脚本，后续脚本依赖前面的全局变量。如果任何一个脚本加载失败（如 CDN 问题、网络故障），后续所有脚本都会报错。
文件：index.html:391-402
✅ 建议：每个模块在顶部添加 typeof X !== 'undefined' 检查；或使用模块加载器。

四、中风险缺陷 (P2 - 影响特定功能或场景)

P2 #10：Brainstorm 对话状态无持久化
根因：Brainstorm 的 messages 数组存储在内存中。如果用户切换 tab 或刷新页面，对话历史全部丢失。只有成功创建书籍后，对话内容才被遗忘。
文件：brainstorm.js:7-18 (state)
✅ 建议：将 Brainstorm 的 messages 存入 sessionStorage，页面刷新后可恢复。
P2 #11：Guide 引导流程状态无持久化
根因：同 Brainstorm，Guide.state 存储在内存中。用户在引导流程中如果返回或刷新，所有进度丢失。
文件：guide.js:6-15 (state)
✅ 建议：将 Guide.state 存入 localStorage（如 guide_state_{bookId}），支持恢复。
P2 #12：书架渲染性能问题 — 每次 renderBookshelf 都遍历全部书籍
根因：app.js:224-267 行，每本书都要调用 Storage.getChapters() 和 Storage.getPlans() 来计算进度。书架有 50+ 本书时，每次渲染触发 50+ 次 localStorage 读取和字符串解析。
文件：app.js:224-267
✅ 建议：在 saveChapters 时同步更新书的进度字段到 books 数组中，渲染时直接读取。
P2 #13：showBookDetail 每次调用都重新解析章节规划正则
根因：app.js:381 行 plans.chapter_plan.match(/第(\d+)章/g) 每次都重新匹配全文，对于大段章节规划文本（如 10K 字），正则匹配开销较大。
文件：app.js:381-398
✅ 建议：在 savePlan 时将解析结果缓存到 plans 对象中。
P2 #14：IndexedDB 无版本迁移机制
根因：ChapterDB 的 DB_VERSION 固定为 1。如果未来需要修改存储结构（如增加字段），需要手动处理迁移。当前没有版本检查和迁移逻辑。
文件：storage.js:17
✅ 建议：在 onupgradeneeded 中增加版本判断和迁移逻辑。
P2 #15：AI API Key 明文存储在 localStorage
根因：虽然 AIService.obfuscateKey() 使用了简单的 XOR 混淆，但密钥和算法都暴露在 JS 源码中。有 root 权限的设备或恶意代码可以直接解密。
文件：ai-service.js:77-89
✅ 建议：使用 Android Keystore / iOS Keychain 存储密钥；或提示用户风险。
P2 #16：pickFile 中的 _pickFileStub 引用未定义
根因：android-bridge.js:253 行检查 window.__onFilePicked !== _pickFileStub，但 _pickFileStub 变量从未定义。虽然不会导致崩溃（undefined 比较），但比较逻辑无意义。
文件：android-bridge.js:253
✅ 建议：修正比较逻辑或删除无用的 _pickFileStub 引用。
