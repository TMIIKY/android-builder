一、系统架构概览

灵墨采用典型的前端单页应用（SPA）+ WebView 混合架构，整体分为四个层次：

关键架构决策

  • 混合存储：localStorage 存元数据（书籍/记忆/设置/规划），IndexedDB 存章节正文大文本。这种分离是合理的——localStorage 同步读写快适合元数据，IndexedDB 异步适合大文本——但增加了数据一致性风险（见缺陷#1、#2）。
  • Spec 驱动生成：每章先生成 YAML Spec（before_state/after_state/must_happen），再基于 Spec 写正文。这是与"开放小说创作助手" Skill 最核心的对齐点。
  • 持久记忆系统：per-book 的人物状态历史、伏笔回收追踪、因果链压缩、世界状态——这是长篇创作质量的关键保障。
  • 多路径桥接：NativeShare → Capacitor Share → plus.io → 浏览器下载，四级降级确保各环境可用。
  • Sheet Overlay + 原生返回栈：每个 Sheet 打开/关闭同步通知原生层栈深度，确保 Android 返回键行为正确。

模块依赖关系

index.html 加载顺序: capacitor-core → capacitor-filesystem → android-bridge → storage → data-backup → ai-service → cover-engine → workflow → plan-manager → guide → brainstorm → app

二、P0 严重缺陷（4个）

P0 #1：章节修改后 IndexedDB 未同步，数据丢失风险
文件：app.js，函数 _doReviseChapter（第1125-1171行）和 _doRegenerateChapter（第1173-1299行）

根因：两个函数都调用了 Storage.saveChapters(currentBookId, chapters)，该函数内部会将 content 从元数据中剥离后异步写入 IndexedDB。但这两个函数紧接着调用 openReader() 刷新阅读器，未等待 IndexedDB 写入完成。如果此时 App 崩溃或页面被关闭，修改后的章节正文会丢失。

// _doReviseChapter 第1160行 target.content = aiResult.trim(); Storage.saveChapters(currentBookId, chapters); // ⚠️ saveChapters 内部异步写 IndexedDB，这里未 await openReader(currentChapterIndex); // 立即刷新阅读器
✅ 建议：改用 Storage._indexedDBPutChapter() 先同步写入 IndexedDB 再刷新；或让 saveChapters 返回 Promise 并 await。
P0 #2：_handleBackAction 无条件触发 Guide 模块，可能导致运行时错误
文件：app.js，第89-94行

根因：只检查了 Guide.state.mode 是否为真值，但未检查 Guide 页面（guidePage）是否正在展示。在用户处于书架/设置页面按返回键时，如果之前使用过 Guide 模块（state.mode 仍为非空），会错误地触发 Guide._goBack()，可能导致页面跳转异常。

// 第89-94行 if (typeof Guide !== 'undefined' && Guide._goBack && Guide.state && Guide.state.mode) { Guide._goBack(); _popNativeStack(); return; }
✅ 建议：增加 guidePage 的 active 状态检查： document.getElementById('guidePage')?.classList.contains('active')
P0 #3：clearAllData() 使用 localStorage.clear() 清除非灵墨数据
文件：app.js，第1689行

根因：localStorage.clear() 会清除所有 localStorage 键值对，包括其他库/插件的数据、浏览器缓存等。在 WebView 混合应用中，可能误删 Capacitor 插件状态或其他 SDK 数据。

// 第1689行 localStorage.clear(); // ⚠️ 清除所有键，非仅灵墨数据
✅ 建议：使用 DataBackup.collectAllData() 的逻辑精确匹配 key 前缀（books/chapters_/memory_/plans_/specs_/reviews_/ai_models/active_model_id/bs_），逐项 remove。
P0 #4：数据导入时 IndexedDB 数据未纳入回滚快照
文件：data-backup.js，_doImport()（第558-629行）

根因：导入备份时只快照了 localStorage 的当前值用于回滚，但未处理 IndexedDB 中的章节正文数据。如果导入中途失败，localStorage 回滚了但 IndexedDB 可能已被部分覆盖，导致数据不一致。

// _doImport 第565-578行 const tempSnapshot = {}; keys.forEach(key => { const existing = localStorage.getItem(key); if (existing !== null) tempSnapshot[key] = existing; }); // ⚠️ 未备份 IndexedDB 中的章节正文
✅ 建议：导入前先收集所有 bookId，备份对应的 IndexedDB 章节数据到临时 ObjectStore；导入失败时恢复 IndexedDB 数据。

三、P1 重要缺陷（5个）

P1 #5：章节 ID 碰撞风险
文件：app.js，第711行

根因：章节 ID 生成使用 Math.random().toString(36).substring(2, 6) 仅产生 4 位随机字符（~1.6M 种可能），配合时间戳在同一毫秒内批量创建多章时碰撞概率不可忽略。

// 第711行 const randomSuffix = Math.random().toString(36).substring(2, 6); const shellId = 'ch_' + baseTimestamp + '_' + randomSuffix + '_' + i;
✅ 建议：使用 crypto.randomUUID() 或增加随机长度到 8 位以上。
P1 #6：封面渲染非确定性导致视觉闪烁
文件：cover-engine.js，_starsHTML()（第173-183行）

根因：_starsHTML() 使用 Math.random() 生成星星位置，每次调用产生不同的位置。当详情页通过 showBookDetail() 重建 DOM 时（先清空再 innerHTML），封面重新渲染导致星星位置跳动。

// 第175-178行 for (let i = 0; i < 10; i++) { const top = Math.random() * 90; // ⚠️ 每次不同 const left = Math.random() * 90; // ⚠️ 每次不同
✅ 建议：基于书名/ID 的 hash 生成确定性的随机种子（如简单的 hash 函数）。
P1 #7：buildContextMemory 大纲截断策略过于激进
文件：plan-manager.js，_buildBatchDetailPrompts()（第1131行）

根因：大纲截断长度根据总章数计算——50章以下仅截 800 字符，200章以上截 4000 字符。对于复杂大纲（如 50 章但设定极丰富），800 字符可能远远不够，AI 无法获取足够的大纲上下文。

// 第1131行 const outlineCutLen = totalChapters <= 50 ? 800 : (totalChapters <= 200 ? 2000 : 4000);
✅ 建议：改为按实际文本长度的百分比截断（如 50%），同时设上下限（800-4000）。
P1 #8：_continueChapterPlan 无限循环风险
文件：plan-manager.js，第1473-1478行

根因：当 totalChapters === 0（大纲和蓝图中都未解析出总章数）时，chapter_count 被设为 lastPlanned + batchSize，这是一个递增的值。每次调用"继续规划"都会生成新的 batchSize，如果 AI 返回失败但未更新 lastPlanned，可能形成无效循环。

// 第1474行 chapter_count: totalChapters > 0 ? String(totalChapters) : String(lastPlanned + this.state.chapterPlanBatchSize), // ⚠️ totalChapters=0 时 chapter_count 每次递增，无终止条件
✅ 建议：当 totalChapters 无法解析时，弹窗让用户手动输入目标章数，而不是自动递增。
P1 #9：章节规划数据导入后 IndexedDB 不一致
文件：data-backup.js，collectAllData()（第299-326行）

根因：collectAllData() 只收集 localStorage 中的数据（通过 key 前缀匹配），但章节正文实际存储在 IndexedDB 中。导出时 metadata 中 chapter.content 已被 saveChapters() 剥离。导入后 chapters 元数据恢复，但正文仍在旧的 IndexedDB 中或丢失。

// collectAllData 只匹配 localStorage keys key.startsWith('chapters_') || // 只有元数据，不含正文
✅ 建议：导出时显式将 IndexedDB 章节正文嵌入备份 JSON；导入时同步写入 IndexedDB。

四、P2 优化建议（5个）

P2 #10：书架进度计算 O(n) 复杂度
文件：app.js，renderBookshelf()（第235-264行）

每本书渲染时都调用 Storage.getChapters() 和 Storage.getPlans() 来计算进度。当书架有大量书籍时，每次渲染都可能触发多次 IndexedDB 读取。

✅ 建议：在 Storage.saveChapters 时同步更新书的进度字段，渲染时直接读取。
P2 #11：详情页不必要的 DOM 重建
文件：app.js，showBookDetail()（第314-317行）

每次进入详情页都执行 detailScreen.innerHTML = '' 清空再重建。对于频繁切换书籍的用户，这会产生不必要的重绘和性能损耗。

✅ 建议：仅在 bookId 变更时重建 DOM，同书不同章节的刷新仅更新 chapterList 部分。
P2 #12：批量章节生成缺少速率限制
文件：app.js，generateChapters()（第736-919行）

循环中逐个调用 AI API 生成章节，无任何延迟。如果用户一次性请求 10 章，会在短时间内连续发送 10 个请求，可能触发 API 提供商的速率限制（429）。

✅ 建议：在每章生成之间增加 1-2 秒延迟（尤其对于免费 API）。
P2 #13：阅读器未使用虚拟滚动
文件：app.js，renderReaderContent()（第593-618行）

章节内容直接通过 innerHTML 渲染全部段落（content.split('\n')），当章节特别长（如 8000+ 字）时，DOM 节点数可能达到数百个，在低端手机上可能出现滚动卡顿。

✅ 建议：使用 CSS content-visibility: auto 或分页渲染。
P2 #14：_generateTitle 正则表达式局限性
文件：app.js，_generateTitle()（第1077-1089行）

标题提取依赖 "第X章 标题" 格式匹配，如果 AI 未在正文第一行写出此格式，则降级到取前100字符中的引号内容或清理标点后的前4个字符。后者经常产生无意义的标题如"夜色如墨"。虽然已有 _doRegenerateChapter 中的 AI 标题生成作为补充，但 generateChapters 中未使用。

✅ 建议：统一使用 AI 生成标题（像 _doRegenerateChapter 中那样），仅以正则匹配作为降级。

五、运行时风险分析

5.1 数据持久化风险（4个场景）

风险场景触发条件影响缓解建议
localStorage 配额耗尽 用户创建大量书籍/章节（元数据累积），或单本书记忆数据过大 _safeSetItem 触发应急清理（清空最老的 chapters_/memory_），可能导致旧书数据丢失 定期提示用户导出备份；实现"归档"功能将不活跃书籍的元数据压缩或迁移到 IndexedDB
IndexedDB 事务超时 写入大量章节数据（30s 超时），设备存储 I/O 繁忙 章节正文写入失败但元数据已更新，出现"章节存在但内容为空"的不一致状态 增加重试机制（指数退避）；事务失败时回滚元数据状态
App 被杀/Crash 时数据丢失 正在批量生成章节时 App 被系统回收或崩溃 已写入 localStorage 的元数据保留，但异步 IndexedDB 写入可能未完成 生成流程增加"断点续传"标记；启动时检查 pending 状态的章节并恢复
备份导入覆盖现有数据 用户误操作导入旧备份 当前数据被覆盖，虽有回滚保护但仅限 localStorage，IndexedDB 不一致 导入前显示备份摘要（时间/书籍数）；导入后自动校验 IndexedDB 完整性

5.2 AI 服务质量风险（5个场景）

风险场景触发条件影响缓解建议
Spec 生成失败导致降级生成质量下降 generateChapterSpec 抛出异常 降级为直接生成（无 Spec 引导），缺少 before_state/after_state 约束，连贯性降低 增加 Spec 生成重试（至少 2 次）；降级时注入简化版 Spec（基于前文自动构建）
AI 未按格式输出标记 AI 未输出【摘要】【新人物】等标记 autoUpdateMemory 无法提取信息，记忆更新不完整 已有 _generateAISummary 异步补充摘要；可增加输出格式校验并在失败时重试
Writer System Prompt 被截断 DeepSeek 等模型对 system prompt 有长度限制（部分模型限制 ~4K tokens） System Prompt 约 90 行（包含 40 遗憾形态），可能被模型截断导致写作原则不完整 监控实际 token 消耗；对不支持长 system prompt 的模型做动态压缩
自动评审循环无限重试 autoWriteChapter 中评审-修订循环（第259行）评分始终不达标 最多 2 次修订后停止（有保护），但会消耗大量 API 调用 增加总时间限制（如 5 分钟内必须完成）；在评分持续不达标时提示用户人工介入
上下文窗口溢出 buildContextMemory 聚合的记忆 + 前文结尾 + 章节规划超过模型上下文限制 AI 可能忽略最早的内容（"遗忘"前面的剧情），导致连贯性问题 增加 token 估算逻辑（每字符 ~0.5 token）；在接近限制时更激进地截断

5.3 原生桥接风险（4个场景）

风险场景触发条件影响缓解建议
返回栈计数不一致 用户快速连续操作 Sheet 打开/关闭，或 WebView 意外刷新 Android 返回键行为异常：按一次返回多级，或按多次无反应 增加栈深度校验（JS 端维护计数与原生同步检查）；定期 resetStack 校准
Base64 编码文件过大导致原生崩溃 导出超长小说（如 500 章 × 3000 字 = 1.5M 字符），Base64 后 ~2MB Java 端 writeTempFile 可能 OOM 或 Intent 数据超限 大文件分片写入；使用 FileProvider 流式传输代替一次性 Base64
文件选择器回调丢失 pickFile 后 WebView 被重建或页面刷新 window.__onFilePicked 回调丢失，文件选择器打开但结果无法传回 JS 增加超时处理（30s 后自动清理回调并提示用户重试）
Capacitor 插件未就绪 应用启动时 Capacitor 初始化未完成即调用 Filesystem API 备份自动恢复失败，用户数据可能无法加载 增加 waitForCapacitor() 就绪检测；设置合理的超时和降级路径

5.4 UI/UX 边缘情况（5个场景）

风险场景触发条件影响缓解建议
批量生成中切换页面 章节正在异步生成时，用户切换到书架或其他页面 _refreshChapterItemInPlace 尝试更新已不在 DOM 中的元素，静默失败；_refreshDetailAfterGenerate 可能在不正确的页面刷新 增加"离开确认"对话框；批量生成期间禁用底部导航
低端设备大文本渲染 在内存 < 2GB 的设备上打开超长章节（> 10K 字） innerHTML 渲染大量 <p> 标签导致页面卡顿甚至 ANR 分页加载；使用 content-visibility: auto CSS 属性
快速连续点击 用户快速点击"生成章节"按钮多次 虽有 _readerLock 保护阅读器，但章节生成入口 generateChapters() 没有防抖保护 增加 _generatingLock 或按钮禁用状态
网络断连时 AI 调用挂起 AI 生成中途网络断开（WiFi 切换 4G 等） fetch() 等待默认超时（浏览器默认可达 90s），用户界面长时间显示"生成中" 已有 120s AbortController 超时，但可增加网络状态监听和更友好的断连提示
多 Tab 同时操作同一本书 用户在浏览器中打开多个 Tab 操作同一 WebView 数据 localStorage 在多个 Tab 间共享，可能导致竞态覆盖 使用 BroadcastChannel 或 storage 事件同步跨 Tab 状态（低优先级，WebView 场景较少见）

七、综合评估

整体评分：B+（良好，有改进空间）

灵墨是一个设计良好的 AI 辅助小说创作工具，核心的 Spec 驱动创作流程和持久记忆系统是其最大的亮点。但混合存储（localStorage + IndexedDB）的一致性问题是最大的技术债务，建议优先修复 4 个 P0 缺陷。

架构亮点

  • Spec 驱动生成：两阶段（Spec → Content）的章节生成方式，是长篇创作质量的关键创新
  • 持久记忆系统：人物状态历史、因果链压缩、多维度矛盾检测，在同类工具中较为少见
  • 多路径降级：从 NativeShare → Capacitor → plus.io → 浏览器下载的四级降级策略，确保了各环境的兼容性
  • Writer System Prompt 设计：悲剧底色 + 40 遗憾形态 + P0 红线规则，体现了对网文创作规律的深入理解
  • 两阶段章节规划：整体蓝图 + 分批详细规划，解决了长篇小说章节规划的"只见树木不见森林"问题

主要技术债务

  • 混合存储一致性：localStorage 和 IndexedDB 之间缺乏事务保证，数据丢失风险是最紧迫的问题
  • 全局状态管理：currentBookId、currentChapterIndex 等全局变量缺乏封装，多模块间状态同步靠"约定"
  • 错误处理不完整：大量 try/catch 块仅 console.warn 而未向用户提供可操作的反馈
  • 测试缺失：无任何单元测试或集成测试，所有质量保障依赖人工验证
